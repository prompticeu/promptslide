# PromptSlide SaaS Architecture Plan — Agent-Powered Slide Generation via An.dev

## Overview

This plan describes how to add AI-powered slide generation to the existing PromptSlide SaaS app (Next.js, auth, DB already in place in a separate repo). The agent runs on [An.dev](https://an.dev), which abstracts Claude agent orchestration and E2B sandbox management. Users chat with an AI agent that writes PromptSlide `.tsx` slides inside a sandbox, while seeing a live preview. Later, users can also directly edit slides in a code editor.

**Why the existing pipeline can't run in-browser:**
1. Tailwind v4 uses `@import "tailwindcss"`, `@source`, `@theme inline`, `@custom-variant` — all processed by `@tailwindcss/postcss` at build time
2. Path aliases (`promptslide` → core TS source, `@/` → project src/) resolved by Vite
3. Virtual entry module (`virtual:promptslide-entry`) generated by the Vite plugin
4. Full dependency tree: React 19, Framer Motion 12, lucide-react, clsx, tailwind-merge

**Conclusion:** User `.tsx` slides must be compiled where Vite runs. The question is how the compiled output reaches the user's browser, and who controls the UI.

## 2. The iframe Problem — and the Solution

### The Problem

The original plan embeds the entire Vite dev server output in an `<iframe>`. But looking at what `SlideDeck` (`slide-deck.tsx`) actually renders, this is problematic:

**What `SlideDeck` renders (lines 142–430):**
- A floating toolbar (view mode buttons, PDF download, fullscreen) — lines 167–220
- Keyboard shortcuts (Arrow keys, F, G, Space) — lines 102–128
- Slide navigation controls (back/forward arrows, slide counter) — lines 324–349
- Grid view (thumbnail overview with click-to-navigate) — lines 354–394
- List view (vertical scroll for PDF export) — lines 397–427
- Fullscreen presentation mode (via `requestFullscreen()`) — lines 51–57
- Print CSS (`@page { size: 1920px 1080px }`) — lines 144–163

**Why embedding this as-is is wrong:**

1. **Double chrome** — The iframe has its own toolbar + nav, AND the SaaS app needs its own. Two competing sets of controls, visually redundant.
2. **Keyboard conflicts** — The iframe captures arrow keys, F, G, Space. These conflict with the SaaS app's own keyboard handling (e.g. chat input, accessibility).
3. **Fullscreen fails in cross-origin iframes** — `requestFullscreen()` inside an iframe from a different origin is blocked unless the `allow="fullscreen"` attribute is set, and even then the UX is wrong (it fullscreens the iframe, not the SaaS app).
4. **Grid view is duplicated** — The SaaS app wants its own slide thumbnail strip/grid, but the iframe already has one that can't be styled or integrated.
5. **No way to deep-link or control** — The SaaS app can't easily get slide metadata, navigate programmatically, or know the current slide without a postMessage bridge.

### The Solution: Refactor `SlideDeck` → Separate Rendering from Chrome

Split the monolithic `SlideDeck` component into composable parts:

```
BEFORE (monolithic):
  SlideDeck = toolbar + keyboard shortcuts + nav controls + slide renderer
              + grid view + list view + fullscreen + print CSS

AFTER (composable):
  SlideRenderer  — Renders a single slide with animation state. No UI chrome.
  SlideThumbnail — Renders a miniaturized slide for grid/thumbnails. No UI chrome.
  SlideDeck      — Unchanged for CLI/local use (backward-compatible)

  Already separate (no changes needed):
  useSlideNavigation — Navigation state hook
  AnimationProvider  — Animation step context
  SlideThemeProvider — Theme context
  SlideErrorBoundary — Per-slide error boundary
```

The good news: **most of the building blocks are already separate**. The `SlideDeck` component is the only monolithic piece. The refactoring is surgical — extract the rendering logic, don't rewrite anything.

### The `/embed` route: What it renders

A new `SlideEmbed` component — headless, controllable via `postMessage`:

```tsx
// packages/cli/src/core/slide-embed.tsx (NEW)
// This is what renders inside the iframe for SaaS use

export function SlideEmbed({ slides }: { slides: SlideConfig[] }) {
  const { currentSlide, animationStep, totalSteps, showAllAnimations,
          advance, goBack, goToSlide } = useSlideNavigation({ slides })

  // Listen for commands from parent (SaaS app)
  useEffect(() => {
    const handler = (e: MessageEvent) => {
      if (e.data?.type === 'navigate')   goToSlide(e.data.slide)
      if (e.data?.type === 'advance')    advance()
      if (e.data?.type === 'goBack')     goBack()
      if (e.data?.type === 'getState')   reportState()
    }
    window.addEventListener('message', handler)
    return () => window.removeEventListener('message', handler)
  }, [goToSlide, advance, goBack])

  // Report state changes to parent
  useEffect(() => {
    window.parent.postMessage({
      type: 'slideState',
      currentSlide,
      totalSlides: slides.length,
      animationStep,
      totalSteps,
      titles: slides.map(s => s.title)
    }, '*')
  }, [currentSlide, animationStep, slides])

  const CurrentSlideComponent = slides[currentSlide]!.component

  // Just the slide. No toolbar. No nav buttons. No keyboard shortcuts.
  return (
    <div className="h-screen w-screen bg-black">
      <AnimationProvider currentStep={animationStep} totalSteps={totalSteps}
                         showAllAnimations={showAllAnimations}>
        <SlideErrorBoundary slideIndex={currentSlide}
                            slideTitle={slides[currentSlide]?.title}>
          <CurrentSlideComponent
            slideNumber={currentSlide + 1}
            totalSlides={slides.length}
          />
        </SlideErrorBoundary>
      </AnimationProvider>
    </div>
  )
}
```

**Key properties:**
- No toolbar, no nav arrows, no keyboard shortcuts, no view mode switching
- Fills the entire viewport (the iframe is scaled by the SaaS app via CSS `transform: scale()`)
- Communicates bidirectionally with parent via `postMessage`
- Reports slide count, current slide, step state — so the SaaS app can build its own nav UI

### The SaaS app owns all UI

The SaaS app's preview panel renders:
1. An `<iframe src="https://{sandbox-id}-5173.e2b.dev/embed">` scaled to fit
2. Its own navigation strip below the iframe (built from `slideState` messages)
3. Its own grid/thumbnail view (either by requesting screenshots from sandbox, or rendering multiple small iframes)
4. Its own fullscreen button (fullscreens the iframe element, not the sandbox page)

### Vite plugin changes for `/embed` route

The Vite plugin (`plugin.mjs`) currently serves `index.html` for `/`. Add a second route:

```javascript
// In plugin.mjs configureServer()
if (req.url === '/embed' || req.url === '/embed.html') {
  const html = await server.transformIndexHtml(req.url, getEmbedHtmlTemplate())
  // ... serve embed.html that loads SlideEmbed instead of full App
}
```

The embed entry module imports `SlideEmbed` + `SlideThemeProvider` instead of `App`, and injects the `deck-config.ts` slides + `theme.ts` theme. This is a small, additive change to the Vite plugin.

### Editor data flow

```
User types in Monaco → debounce (300-500ms) → write to sandbox via An.dev file API
                                                      │
                                                      ▼
                                              Vite detects file change
                                                      │
                                                      ▼
                                              HMR update → iframe updates
```

**Latency:** The round-trip (browser → SaaS API → An.dev → E2B sandbox → Vite HMR → iframe) will be ~200-500ms. This is fine for a "save and preview" workflow (similar to CodeSandbox/StackBlitz). Not keystroke-by-keystroke, but acceptable.

**File tree:** The SaaS app reads the file list from the sandbox (via An.dev file API) and displays it in a sidebar. Only editable source files are shown (`src/slides/`, `src/layouts/`, `src/deck-config.ts`, `src/theme.ts`, `src/globals.css`).

### Why not Vite alternatives?

The user asked if we should switch away from Vite. The answer is **no, keep Vite**, because:

1. **The compilation problem is the same regardless of bundler.** Tailwind v4 PostCSS, path aliases, React JSX transform — any bundler needs to handle these. Vite already does.
2. **HMR is critical.** Vite's HMR is battle-tested and works perfectly with the existing `promptslide` setup. Switching to esbuild (no HMR), Turbopack (Next.js-only), or Parcel (less ecosystem) would be a downgrade.
3. **The real issue isn't Vite, it's the `SlideDeck` monolith.** The architecture problem is that `SlideDeck` mixes rendering with UI chrome. Refactoring `SlideDeck` solves this regardless of bundler.
4. **Zero migration cost.** Everything already works with Vite. Switching would require rewriting the config, plugin, and build commands.

If a bundler change were considered in the future, the most interesting option would be **Sandpack** (CodeSandbox's in-browser bundler) which can run Vite in a web worker, eliminating the sandbox for preview entirely. But this has unproven Tailwind v4 support and would be a Phase 3+ experiment.

## 6. postMessage Protocol

### SaaS → iframe (commands)

```typescript
// Navigate to specific slide
iframe.contentWindow.postMessage({ type: 'navigate', slide: 3 }, '*')

// Advance one step/slide
iframe.contentWindow.postMessage({ type: 'advance' }, '*')

// Go back one step/slide
iframe.contentWindow.postMessage({ type: 'goBack' }, '*')

// Request current state
iframe.contentWindow.postMessage({ type: 'getState' }, '*')

// Set view mode (for thumbnail generation)
iframe.contentWindow.postMessage({ type: 'setViewMode', mode: 'grid' }, '*')
```

### iframe → SaaS (events)

```typescript
// Emitted on every state change
{
  type: 'slideState',
  currentSlide: 2,          // 0-indexed
  totalSlides: 8,
  animationStep: 1,
  totalSteps: 3,
  titles: ['Title', 'Problem', 'Solution', ...]
}

// Emitted when Vite HMR updates the slides
{
  type: 'hmrUpdate',
  updatedFiles: ['src/slides/slide-market.tsx']
}

// Emitted when an error boundary catches
{
  type: 'slideError',
  slideIndex: 2,
  error: 'Cannot read property...'
}
```

## 8. Sandbox Lifecycle

### 8.1 Pre-warmed Template

E2B template `promptslide-v1` contains:
1. Node.js + bun
2. Scaffolded project with `bun install` completed (node_modules cached)
3. `sandbox-setup.sh` for per-session initialization

Cold start: ~3-5s (vs ~30s without template).

### 8.2 Session Flow

```
User opens/creates presentation → SaaS backend → An.dev: create sandbox
  → E2B provisions from template
  → sandbox-setup.sh:
      1. Set env vars (PROMPTSLIDE_SANDBOX=1, PROMPTSLIDE_HMR_HOST=...)
      2. If resuming: restore source_snapshot files to src/
      3. Apply theme config (write theme.ts, globals.css)
      4. Start promptslide studio --port=5173 &
  → Return sandbox_id + preview_url
  → Frontend loads iframe (/embed) + connects chat (thread_id)
```

### 8.3 Persistence (Hibernation/Resume)

E2B sandboxes are ephemeral (max ~24h). Before hibernation:
1. Read source files: `src/slides/*.tsx`, `src/layouts/*.tsx`, `src/theme.ts`, `src/deck-config.ts`, `src/globals.css`, `src/App.tsx`, `public/*`
2. Store as `source_snapshot` JSONB in DB (~50-200KB per deck)
3. Terminate sandbox

On resume: create new sandbox from template, write snapshot files, start Vite.

### 8.4 Inactivity Management

```
Active sandbox ──(5 min idle)──► Warning toast
    ──(10 min total)──► Auto-snapshot + terminate
    ──(User returns)──► New sandbox, restore snapshot (~3-5s)
```

## 11. Export Pipeline

### PDF Export

Open sandbox URL at `/` (full SlideDeck) with `?view=list` in headless Chrome. The existing print styles (`@page { size: 1920px 1080px }`) format each slide as a page. Call `page.pdf()` → upload to S3/R2.

Note: PDF export uses the full `/` route (with `SlideDeck`), not `/embed`. This is intentional — the full route has the list view and print CSS needed for export.

### Static HTML Export

Run `promptslide build` in sandbox, read `dist/`, package as `.zip`.

## 14. Technical Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| E2B sandbox cold start | Poor UX on first load | Pre-warmed template (~3-5s); loading animation |
| HMR WebSocket through E2B | Preview doesn't update | Correct `server.hmr` config; fallback: iframe reload on file change event |
| Code editor latency | Slow preview updates on edit | Debounce writes (300ms); show "saving..." indicator; Vite HMR is fast once file is written |
| postMessage security | XSS via untrusted messages | Validate message origin; only accept known message types |
| Agent writes invalid TSX | Preview crashes | `SlideErrorBoundary` catches per-slide; `hmrUpdate` event lets SaaS show error state |
| Sandbox timeout | User loses work | Auto-snapshot on inactivity; clear status indicator |

## 15. Architectural Decisions Summary

1. **Refactor `SlideDeck` into composable parts** — The monolithic component mixing rendering with UI chrome is the root cause of the iframe problem. Separating them enables both the SaaS embed use case and future code editor support.

2. **Keep Vite** — The issue was never Vite. It was the `SlideDeck` component owning all the chrome. Vite's HMR, PostCSS pipeline, and plugin system are exactly what's needed.

3. **`/embed` route for SaaS, `/` for local/export** — Two entry points, same compilation pipeline. No code duplication for rendering logic (`SlideRenderer` is shared).

4. **SaaS app owns all UI chrome** — Navigation, thumbnails, fullscreen, view modes are all native SaaS components. The iframe is purely a rendering surface.

5. **postMessage protocol for iframe control** — Clean contract between SaaS app and sandbox. No tight coupling. Works across origins.

6. **Code editor writes to sandbox, not compiles in-browser** — Accept the ~200-500ms round-trip. It's good enough (CodeSandbox/StackBlitz have similar latency). Avoids building a browser-side Tailwind v4 + PostCSS pipeline.

7. **An.dev for agent orchestration** — Abstracts Claude + E2B. SaaS app just makes API calls.
