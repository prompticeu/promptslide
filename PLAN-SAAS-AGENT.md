# PromptSlide SaaS Architecture Plan — Agent-Powered Slide Generation via An.dev

## Overview

This plan describes how to add AI-powered slide generation to the existing PromptSlide SaaS app (Next.js, auth, DB already in place in a separate repo). The agent runs on [An.dev](https://an.dev), which abstracts Claude agent orchestration and E2B sandbox management. Users chat with an AI agent that writes PromptSlide `.tsx` slides inside a sandbox, while seeing a live preview via an embedded iframe connected to the sandbox's Vite dev server.

**Key principle:** Reuse the existing PromptSlide rendering pipeline (Vite + React 19 + Tailwind v4 + Framer Motion) running inside the sandbox. No client-side compilation, no parallel rendering engine.

```
├── SKILL.md                     — Full authoring guide (~10KB)

    └── assets/templates/        — 8 reference slide templates (.tsx)

```

**Why the existing pipeline can't run in-browser:**
1. Tailwind v4 uses `@import "tailwindcss"`, `@source`, `@theme inline`, `@custom-variant` — all processed by PostCSS at build time
2. Path aliases (`promptslide` → core TS source, `@/` → project src/) resolved by Vite
3. Virtual entry module (`virtual:promptslide-entry`) generated by the Vite plugin
4. Full dependency tree: React 19, Framer Motion 12, lucide-react, clsx, tailwind-merge

**Conclusion:** The Vite dev server must run in a sandbox. The browser gets an iframe.

## 2. Rendering Approach: Vite-in-Sandbox + iframe

### How It Works

1. An.dev provisions an E2B sandbox with a pre-warmed template (node_modules pre-installed, project scaffolded)
2. `promptslide studio --port=5173` runs inside the sandbox
3. The SaaS frontend embeds the sandbox's forwarded port in an `<iframe>`
4. When the agent writes/modifies `.tsx` files, Vite's file watcher triggers HMR → the iframe updates instantly
5. The user sees slides appear and change in real-time as the agent works

### Why This Is Right

- **Zero fidelity loss** — same rendering as local development
- **HMR for free** — Vite already supports instant hot-reload
- **No maintenance burden** — no parallel rendering engine to keep in sync
- **Error handling exists** — `SlideErrorBoundary` (already in `slide-deck.tsx`) catches broken slides gracefully

### Critical Vite Config for Sandbox

The `createViteConfig()` in `packages/cli/src/vite/config.mjs` needs sandbox-aware server settings:

```javascript
// Additional server config when running inside E2B sandbox
server: {
  port: 5173,
  host: '0.0.0.0',           // Listen on all interfaces (required for E2B port forwarding)
  strictPort: true,
  cors: true,                 // Allow cross-origin iframe embedding
  hmr: {
    host: `${sandboxId}-5173.e2b.dev`,  // External hostname for WebSocket
    clientPort: 443,          // E2B forwards via HTTPS
    protocol: 'wss'           // Secure WebSocket through E2B proxy
  }
}
```

**This is the single most critical config detail.** Without correct HMR settings, the initial page loads in the iframe but never updates when files change.

**Implementation approach:** Add an environment variable check (e.g. `PROMPTSLIDE_SANDBOX=1` + `PROMPTSLIDE_HMR_HOST=...`) in `createViteConfig()` so the same code works both locally and in sandbox mode. The sandbox setup script sets these env vars before starting the dev server.

## 4. Sandbox Lifecycle

### 4.1 Pre-warmed Template

Create an E2B sandbox template (`promptslide-v1`) with:

1. Node.js + bun installed
2. `npx promptslide create my-deck --non-interactive` already run
3. `bun install` completed (node_modules cached in the image)
4. A `sandbox-setup.sh` script for per-session initialization

This reduces cold start from ~30s (full install) to ~3-5s (just apply user state + start Vite).

### 4.2 Session Initialization Flow

```
User opens/creates presentation in SaaS app
    │
    ▼
SaaS backend → An.dev API: create sandbox from "promptslide-v1" template
    │
    ▼
An.dev provisions E2B sandbox
    │
    ▼
Run sandbox-setup.sh:
  1. Set env vars (PROMPTSLIDE_SANDBOX=1, PROMPTSLIDE_HMR_HOST=<sandbox-url>)
  2. If resuming: restore source files from DB snapshot → write to src/
  3. If new: project is already scaffolded from template
  4. Apply user's theme config if provided (write theme.ts, globals.css)
  5. Start: promptslide studio --port=5173 &
    │
    ▼
Return sandbox_id + preview_url to SaaS frontend
    │
    ▼
Frontend loads iframe (preview_url:5173) + connects An.dev chat (thread_id)
```

### 4.3 Session Persistence (Hibernation/Resume)

E2B sandboxes are ephemeral (max ~24h). For long-term storage:

**Before hibernation** (triggered by inactivity or explicit save):
1. Read source files via sandbox filesystem: `src/slides/*.tsx`, `src/layouts/*.tsx`, `src/theme.ts`, `src/deck-config.ts`, `src/globals.css`, `src/App.tsx`, `public/*`
2. Store as `source_snapshot` JSONB in the presentations table (~50–200KB per deck)
3. Terminate sandbox

**On resume:**
1. Create new sandbox from pre-warmed template
2. Write all `source_snapshot` files to sandbox filesystem
3. Write any user-uploaded assets from object storage
4. Start `promptslide studio`
5. Resume the An.dev thread for conversation continuity

### 4.4 Inactivity Management

```
Active sandbox ──(5 min no interaction)──► Warning toast in UI
    ──(10 min total)──► Auto-snapshot + terminate sandbox
    ──(User returns)──► Create new sandbox, restore snapshot (~3-5s)
```

### 6.3 Preview iframe Communication

The SaaS frontend communicates with the Vite dev server in the iframe via `postMessage` for:
- **Slide navigation**: Send `{ type: 'navigate', slide: 3 }` to jump to a specific slide
- **View mode**: Send `{ type: 'viewMode', mode: 'slide' | 'grid' | 'list' }`
- **Slide count**: Receive `{ type: 'slideCount', count: 12 }` for the nav strip

This requires a small `postMessage` listener added to the virtual entry module in `plugin.mjs` (or injected via the sandbox setup). This is a minor, additive change.

### 6.4 Theme Customizer

A UI panel (drawer or modal) where users set brand parameters before/during agent conversation:
- Company name (text input)
- Brand color (color picker → OKLCH value for `--primary`)
- Logo upload (saved to sandbox `public/` + object storage for persistence)
- Font selection (dropdown)

On change, the customizer writes updated `theme.ts` and `globals.css` to the sandbox via the An.dev file API. Vite HMR updates the preview automatically.

## 8. Export Pipeline

### 8.1 PDF Export

Two approaches, both leveraging existing functionality:

**Option A — Print CSS (recommended, simplest):**
1. Open sandbox Vite URL with `?view=list` in headless Chrome (Puppeteer/Playwright)
2. The existing print styles in `slide-deck.tsx` (lines 146–163) set `@page { size: 1920px 1080px }` and format each slide as a page
3. Call `page.pdf()` → high-quality PDF with correct dimensions
4. Upload to S3/R2, return download URL

**Option B — Screenshot-based:**
1. Run `promptslide build` in sandbox via exec API
2. Open built SPA in headless Chrome
3. Screenshot each slide at 1920×1080
4. Combine into PDF

Option A is preferred because it reuses the print CSS that already exists and doesn't require a production build step.

**Where to run headless Chrome:**
- Browserless.io (managed service) — simplest
- Self-hosted Puppeteer on a small worker (Fly, Railway)
- Inside the E2B sandbox itself (if Chromium is in the template)

### 8.2 Static HTML Export

1. Run `promptslide build` in sandbox via An.dev exec API
2. Read `dist/` directory via file API
3. Package as `.zip`
4. Upload to S3/R2
5. Optionally: deploy to a preview URL (Vercel, Cloudflare Pages)

```javascript
export function createViteConfig({ cwd, mode = "development" }) {

      port: 5173,

      strictPort: true,
      cors: true,
      hmr: hmrHost ? {
        host: hmrHost,
        clientPort: 443,
        protocol: 'wss'
      } : true
    } : undefined
  }
}
```

### 9.2 PostMessage Bridge (Optional)

**File:** `packages/cli/src/vite/plugin.mjs`

Add a small postMessage listener to the virtual entry module so the parent iframe can control navigation:

```javascript
// Injected into the entry module when PROMPTSLIDE_SANDBOX=1

    window.dispatchEvent(new CustomEvent('promptslide:navigate', { detail: e.data }))

})
```

The `SlideDeck` component would listen for this custom event. This is additive — no breaking changes to local usage.

## 12. Technical Risks & Mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| E2B sandbox cold start (~20-30s without template) | Poor UX on first load | Pre-warmed template reduces to ~3-5s; show loading animation |
| HMR WebSocket fails through iframe | Preview loads but doesn't update | Correct `server.hmr` config (§2); fallback: periodic iframe reload |
| CORS blocks iframe embedding | Preview doesn't load at all | `server.cors: true` + correct E2B port forwarding headers |
| Agent writes invalid TSX | Preview crashes | `SlideErrorBoundary` already exists in `slide-deck.tsx` — catches per-slide errors gracefully |
| Sandbox timeout (E2B max ~24h) | User loses work mid-session | Auto-snapshot on inactivity (§4.4); clear status indicator |
| Large source_snapshot | DB performance degradation | Typical decks are 50-200KB; set 50-slide limit; compress if needed |
| An.dev API latency | Slow agent responses | SSE streaming shows partial responses; non-blocking UI |

## 13. Key Architectural Decisions Summary

1. **Vite-in-sandbox + iframe** — Reuses the entire rendering pipeline. Zero fidelity loss. HMR provides real-time updates. No parallel rendering engine needed.

2. **No client-side compilation** — Tailwind v4 + PostCSS + virtual modules + path aliases are too complex for in-browser compilation. Would create a fragile parallel system.

3. **An.dev as orchestration layer** — Abstracts Claude agent SDK + E2B sandbox management. Reduces infrastructure complexity in the SaaS app to API calls.

4. **Source snapshot persistence** — Lightweight JSON of source files (~50-200KB). Exact reproducibility on sandbox restore. No need to persist node_modules or build artifacts.

5. **Pre-warmed sandbox templates** — Eliminate install time. Template has node_modules cached. Per-session setup is just file writes + start Vite.

6. **Existing skill docs transfer directly** — The 30KB+ of agent documentation in `skills/promptslide/` works with An.dev's skill system with minimal adaptation (remove scaffolding sections).

7. **Existing registry powers template marketplace** — CLI already has `publish`/`add`/`search`/`list` commands pointing at `registry.promptslide.dev`.
